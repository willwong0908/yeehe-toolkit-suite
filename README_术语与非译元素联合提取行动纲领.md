# 术语与非译元素联合提取行动纲领

版本：v0.3 共识稿

状态：根据多轮用户反馈修订，作为当前实现方向的需求共识稿。

## 1. 目标

工具需要同时支持“术语提取”和“非译元素提取”。其中，只要用户选择提取术语，就默认包含非译元素提取和非译元素保护流程。工具不再提供“纯术语但完全不处理非译元素”的模式。

运行模式调整为两种：

1. 提取术语：包含非译元素提取、正则整理、全文保护、数值归一、清洗文本去重、术语召回和术语校验。
2. 仅提取非译元素：只发现非译元素、生成/整理正则、排序，并导出排好序的 `非译元素正则` sheet，不进入术语召回和术语校验。

非译元素提取不是附属功能，而是术语提取的前置清洗能力。它负责识别并保护 tag、变量、占位符、格式控制符、程序片段等内容，把这些内容替换成简单占位 tag，降低术语召回的噪声和 token 消耗。

## 2. 核心原则

1. 术语和非译元素分开建模、分开配置、分开输出。
2. 非译元素优先通过正则表达式稳定识别，AI 负责发现未知样式、生成缺失正则，并对最终非译元素正则执行顺序做排序。
3. 正则顺序分为两类：宽泛检测正则的顺序由用户控制，工具不干预；最终非译元素正则的执行顺序由 AI 排序。
4. 术语召回使用清洗后的文本，但输出仍保留原始来源、原始上下文和证据。
5. 数值类差异可以进入清洗权重体系，用于把“除数值外完全一致”的文本归并为同一召回文本。
6. 所有 AI 参与的步骤都要尽量批量化、去重，避免重复消耗 token。

## 3. 参考现有 tag 预处理工具

参考目录：

`D:\脚本\tag预处理`

优先参考这些文件：

1. `tag_ai_core.py`：tag 分类、标准化、AI 生成正则、AI 正则排序。
2. `builtin_regex_library.json`：内置正则库。
3. `regex_config.json`：用户正则配置、启用状态、执行顺序。
4. `excel_tag_detector.py`：正则管理 UI、拖拽排序、检测结果展示、导出思路。

迁移原则：

1. 只吸收成熟逻辑，不直接搬旧 UI。
2. 内置正则库可以迁移为新项目的 `nontrans_builtin_regex_library`。
3. AI 正则生成和正则排序可以复用思路，但需要接入当前工具的 provider、scheduler、runtime。
4. 旧项目里的乱码命名、历史包、build/dist 产物不作为实现依据。

## 4. 关键概念

### 4.1 术语

术语是需要进入术语库、后续翻译时保持一致的游戏概念。

例子：

1. 角色名。
2. 技能名。
3. 道具名。
4. 地点名。
5. 系统名。
6. 属性名。

### 4.2 非译元素

非译元素是翻译或术语提取时不应该被当作自然语言处理的结构化片段。

例子：

1. `<color=#FFAA00>`
2. `</color>`
3. `{0}`
4. `{PlayerName}`
5. `${item.name}`
6. `%s`
7. `\n`
8. `[sprite=icon_coin]`
9. `&nbsp;`
10. 程序变量、资源 key、格式控制符。

### 4.3 正则规则

每条非译元素正则规则在内部至少包含：

| 字段 | 含义 |
|---|---|
| `rule_id` | 稳定 ID |
| `name` | 规则名称 |
| `pattern` | 总匹配正则 |
| `open_pattern` | 开始标签正则，可为空 |
| `close_pattern` | 结束标签正则，可为空 |
| `empty_pattern` | 独立/自闭合/占位符正则，可为空 |
| `element_type` | 非译元素类型，例如 html、brace、bracket、variable、escape、format、program_token |
| `source` | 内部来源标记，仅用于调试和合并，不导出到最终 sheet |
| `enabled` | 内部启用状态，不导出到最终 sheet |
| `order_index` | 执行顺序 |
| `examples` | 命中的非译元素例子 |
| `conflict_notes` | 内部排序或覆盖冲突说明，不导出到最终 sheet |

## 5. 联合运行总流程

联合运行流程如下：

```text
读取源文件
↓
收集源文本
↓
按用户配置的宽泛 ASCII / tag 正则筛出疑似包含非译元素的文本条目
↓
按条目边界切块，发给 AI 召回非译元素
↓
汇总 AI 返回的非译元素
↓
用内置正则库匹配这些非译元素
↓
对内置正则无法覆盖的非译元素，请 AI 生成正则、类别、开始/结束/空类型
↓
正则去重、校验
↓
最终非译元素正则交给 AI 排序
↓
对全量文本执行非译元素保护，把命中片段替换成简单占位 tag
↓
数值差异归一，生成术语召回清洗文本
↓
清洗文本去重
↓
按条目边界切块，发给 AI 召回术语
↓
程序校验术语必须逐字命中原始文本或可追溯清洗文本
↓
术语第二轮校验
↓
导出术语结果、待审核结果、失败记录、非译元素正则表
```

## 6. 非译元素提取流程

### 6.1 宽泛条目筛选

工具先用用户可配置的宽泛正则，从所有文本中筛出“疑似包含非译元素的文本条目”。这个阶段不是为了把非译元素片段先抠出来，而是为了决定哪些整条文本需要发给 AI。

这个阶段的正则顺序完全由用户决定，工具除了提供默认顺序外，不自动调整、不用 AI 调整，也不允许内置正则库抢先改写文本。

重要约束：

1. 宽泛检测阶段只筛条目，不切割、不替换、不生成最终非译元素。
2. AI 接收的是整条文本条目，而不是宽泛正则抠出来的候选片段。
3. 例如 `<color={1}>自动采集器</color>` 这一条会作为完整文本交给 AI，AI 自己判断 `<color={1}>`、`</color>`、`{1}` 是否属于非译元素。
4. 内置正则库不参与这个阶段的切割和替换，避免把完整结构拆坏。
5. 工具只保存用户顺序，并按这个顺序筛选条目和记录命中信息。

默认宽泛方向：

1. 尖括号片段：`<...>`
2. 花括号片段：`{...}`
3. 方括号片段：`[...]`
4. 美元变量：`$...`
5. 百分号格式符：`%s`、`%d`、`{0}%` 等需要进一步定义。
6. 转义字符：`\n`、`\t`、`\uXXXX`
7. HTML 实体：`&nbsp;`
8. 连续 ASCII 程序片段。

用户应可以编辑：

1. 新增/修改/删除宽泛扫描正则。
2. 对正则进行排序


### 6.2 AI 召回非译元素

把疑似包含非译元素的文本条目按设定字符数切块，切块必须以条目为边界。AI 看到的是整条文本，不是正则预先截出来的候选片段。

要求 AI 返回：

```json
{
  "items": [
    {
      "id": "1",
      "element": "<color=#FFAA00>",
      "element_type": "html"
    }
  ]
}
```

第一阶段只要求返回非译元素本体和类型，不要求 AI 直接写正则。原因是正则生成比元素发现更难，必须分阶段做。

### 6.3 内置正则覆盖判断

AI 返回所有非译元素后，程序再用内置正则库逐个匹配，用来判断哪些非译元素已经被现有规则覆盖，哪些需要 AI 生成新正则。

内置正则库在这里只做“覆盖判断”，不做 AI 前预替换，也不改变宽泛检测阶段得到的候选文本。

如果某个非译元素已被内置正则覆盖：

1. 记录命中规则。
2. 记录样例。
3. 不再要求 AI 为它生成新正则。

如果某个非译元素没有被内置正则覆盖：

1. 进入“缺失正则生成池”。
2. 按 normalized element 去重。
3. 相同或等价样式只生成一次正则。

### 6.4 AI 生成缺失正则

对缺失正则的非译元素分组后，请 AI 返回：

```json
{
  "rules": [
    {
      "name": "html_color_tag",
      "pattern": "<color\\s*=\\s*[^<>]+>",
      "open_pattern": "<color\\s*=\\s*[^<>]+>",
      "close_pattern": "",
      "empty_pattern": "",
      "element_type": "html",
      "examples": ["<color=#FFAA00>", "<color = {1}>"]
    }
  ]
}
```

程序必须校验：

1. 正则能编译。
2. 正则至少能匹配来源样例。
3. 正则不能过宽到匹配大量普通文本。
4. 正则不能包含危险或明显灾难性回溯结构。
5. AI 返回的类型必须在允许集合内。

### 6.5 正则去重

正则去重不能只看字符串完全相同。

至少需要：

1. `pattern` 完全相同：合并。
2. `open/close/empty` 完全相同：合并。
3. 对同一批样例的匹配集合完全相同：标记为疑似重复。
4. AI 生成规则如果被已有规则完全覆盖，不新增，只把样例补到已有规则。

## 7. 正则排序原则

正则排序分为两种完全不同的情况，不能混在一起。

### 7.1 宽泛检测正则排序

宽泛检测正则用于“筛出疑似包含非译元素的文本条目”。这个阶段的排序规则：

1. 用户自己排序。
2. 工具只提供默认排序。
3. 默认排序参考 `D:\脚本\tag预处理\regex_config.json` 中的 `pattern_order`。
4. 工具永远不自动调整这个顺序。
5. AI 永远不调整这个顺序。

原因：

1. 宽泛检测阶段只是决定哪些条目需要进入 AI 非译元素召回。
2. 工具不能把宽泛正则命中的片段当成最终非译元素。
3. 工具不能在这个阶段因为内置规则命中 `{1}` 就把 `<color={1}>` 拆坏。
4. 这个阶段尊重用户意图比“智能排序”更重要。

### 7.2 最终非译元素正则排序

最终非译元素正则用于“全量文本保护”。这个阶段的排序规则：

1. 永远交给 AI 排序。
2. 工具负责把规则、样例、覆盖风险、open/close/empty 信息整理给 AI。
3. 工具只校验 AI 返回的排序是否包含全部规则且没有重复。
4. 工具不擅自改写 AI 的最终排序。

最终正则排序的核心目标：

> 更具体、更长、更完整的规则排前面；更宽泛、更短、更容易吞噬其它规则的规则排后面。

用户例子：

```text
规则 A：<color = {1}>
规则 B：{1}
```

如果 B 先执行，`{1}` 会先被替换，A 就无法完整命中。所以 A 必须排在 B 前面。

AI 排序时需要遵守：

1. 如果规则 A 在样例中经常完整包含规则 B 的命中范围，则 A 排在 B 前面。
2. 如果规则 A 的平均命中长度明显大于 B，且 A/B 有重叠，则 A 优先。
3. 如果规则 A 是带外壳的结构，例如 `<...{...}...>`，B 是内部占位符，例如 `{...}`，A 优先。
4. 如果一条规则是宽泛兜底规则，例如 `<[^<>]+>`、`\{[^{}]+\}`，它应该排在更具体的 HTML、变量、占位符规则之后。
5. open/close/empty 三类规则都参与全局排序，不能只在各自小类内部排序。

AI 排序输出建议：

```json
{
  "ordered_ids": ["rule_003", "rule_001", "rule_002"],
  "reasons": [
    {
      "rule_id": "rule_003",
      "reason": "matches longer enclosing color tags before inner placeholders"
    }
  ]
}
```

## 8. 全量文本非译元素保护

排序完成后，对全量文本执行正则保护。

例子：

```text
原文：<color>自动采集器</color>
保护后：<1>自动采集器<2>

原文：{tag} 自动采集器
保护后：<1>自动采集器
```

每条文本都需要保存映射：

| 占位 tag | 原始非译元素 | 规则 ID | 类别 | 开始位置 | 结束位置 |
|---|---|---|---|---:|---:|
| `<1>` | `<color>` | rule_001 | html_tag | 0 | 7 |
| `<2>` | `</color>` | rule_002 | html_tag | 13 | 21 |

注意：

1. 占位 tag 只用于内部清洗和召回。
2. 术语评分和重复判断时，占位 tag 不计分。
3. 输出时仍需要保留原始非译元素。
4. 如果正则命中发生重叠，默认保留先命中的规则，跳过后续重叠命中。
5. 每条文本内部的占位 tag 编号都从 `<1>` 重新开始。

## 9. 数值差异归一

用户提出的需求：

```text
暴击伤害提高20%
暴击伤害提高30%
```

这类文本除了数值外完全一致，应该被当作重复或近似重复。只要存在这种重复项，就把数值清掉：

```text
暴击伤害提高20% -> 暴击伤害提高
暴击伤害提高30% -> 暴击伤害提高
```

已确认第一版做成“条件触发数值清洗”：

1. 先生成一个保守的 numeric-normalized key。
2. 如果多个文本在移除数值后完全相同，则这一组触发数值清洗。
3. 触发后，这组文本用于术语召回的文本去掉数值。
4. 如果某条文本没有形成数值差异组，默认不强制删除数值，避免误删 `T11`、`7日签到`、`3星角色` 等可能有语义的内容。

默认数值候选：

1. 整数、小数、负数。
2. 百分比：`20%`
3. 加减号数值：`+20`
4. 乘数：`x2`、`2x`
5. 范围：`10-20`
6. 时间单位、数量单位在形成数值差异重复组时，跟随数值一起清理；没有形成重复组时不清理。

## 10. 清洗文本去重

术语召回前，每条文本会生成多个版本：

| 版本 | 用途 |
|---|---|
| `original_text` | 原始证据和导出 |
| `nontrans_protected_text` | 非译元素替换成 `<1>`、`<2>` 后的文本 |
| `term_recall_clean_text` | 非译 tag 不计分、数值按规则移除后的召回文本 |
| `dedupe_key` | 用于判断是否重复送 AI |

例子：

```text
row 12: <color>自动采集器</color>
row 58: 自动采集器
row 91: {tag} 自动采集器
```

保护后：

```text
row 12: <1>自动采集器<2>
row 58: 自动采集器
row 91: <1>自动采集器
```

用于去重时，tag 不计分：

```text
row 12 dedupe_key: 自动采集器
row 58 dedupe_key: 自动采集器
row 91 dedupe_key: 自动采集器
```

因此只需要送 AI 一次。

## 11. 术语召回切块优化

术语召回不应继续以“每条 item 必须返回结果”的方式为主。

目标方式：

1. 使用 `term_recall_clean_text`。
2. 先按 `dedupe_key` 去重。
3. 按条目边界累计切块。
4. 每个 chunk 让 AI 返回全局术语列表，而不是每个 id 的 surface_forms。
5. 程序把术语回填到所有包含该术语的来源条目。

切块原则：

1. 普通条目不能从中间截断。
2. 如果单条文本超过限制，才按标点和安全边界切分。
3. chunk 里保留轻量行号，方便 debug，但不要求 AI 逐行返回。
4. chunk 输出只要术语列表，减少空数组和 id 对齐成本。

## 12. 术语校验流程

术语校验阶段只做“候选词是否应该进入术语库”的语义裁决，不再让 AI 输出不可靠的置信度，也不让程序生成语义风险标签去引导 AI。

核心原则：

1. 程序只做确定性处理，不做语义判断。
2. 程序不做 hard approve，不直接批准候选进入正式术语库。
3. 程序不生成、不传递 `risk_hints`、`risk_flags`、`review_priority` 等风险标签。
4. AI 不返回 `confidence`，不返回 `evidence_text`，不返回风险标签。
5. 泛词、普通动词、副词、残片、短词、单次出现等语义问题，通过提示词约束 AI 自己判断。
6. 非译元素内部噪声主要由前置非译元素保护流程排除，校验阶段只保留必要的确定性兜底。

程序负责：

1. 过滤空字符串、纯数字、纯百分比、纯符号等确定性无效候选。
2. 校验候选必须能逐字追溯到来源文本或清洗文本。
3. 过滤明显非译元素残留和明显程序 key。
4. 合并相同候选，统计出现次数和来源。
5. 选择 1 到 3 条代表上下文，并压缩到候选附近。
6. 校验 AI 返回 JSON 格式、`id`、`surface_form`、`decision`、`term_type` 是否合法。
7. 根据 AI 的 `decision` 和用户输出策略，把结果分流到术语库、待审核或剔除。

程序不负责：

1. 判断某个词是否为泛词。
2. 判断某个词是否为普通动词、副词或说明词。
3. 判断某个词是否为更长术语的残片。
4. 判断单次出现的词是否应该成为术语。
5. 给 AI 传递任何语义风险提示。

AI 校验输入只包含客观材料：

```json
{
  "items": [
    {
      "id": "1",
      "surface_form": "自动采集器",
      "occurrence_count": 3,
      "contexts": [
        "解锁纳米打印机配方：<1>自动采集器<2>"
      ]
    }
  ]
}
```

AI 校验输出固定为：

```json
{
  "items": [
    {
      "id": "1",
      "surface_form": "自动采集器",
      "decision": "approved",
      "term_type": "道具",
      "reason": "作为可解锁配方名称出现，是稳定游戏概念。"
    }
  ]
}
```

字段约束：

| 字段 | 说明 |
|---|---|
| `decision` | 只能是 `approved`、`review`、`rejected` |
| `term_type` | `approved/review` 时必须是启用的术语类型；`rejected` 时必须是 `non_term` |
| `reason` | 简短说明，不需要长篇解释 |

结果分流：

```text
AI rejected → 剔除
AI review → 待审核
AI approved + 输出策略允许 → 术语库
AI approved + 输出策略要求保守 → 待审核
```

单次出现策略作为输出策略处理，不传给 AI 做风险提示。默认建议：单次出现的候选即使 AI 判定 `approved`，也先进入待审核；后续可在设置中允许直接入库。

## 13. 输出设计

仅提取非译元素时，输出只需要一张 sheet：

1. `非译元素正则`

这张 sheet 不是旧工具里的“检测命中结果导出”，也不是把旧工具内部对象字段照搬出来。它只保留用户真正需要使用和审核的最终正则结果。

参考 `tag预处理` 项目中 AI 生成正则后的处理逻辑：

1. `expand_regex_rows()` 将每个 tag 家族的 `open`、`close`、`empty` 展开成可执行正则行。
2. `deduplicate_ai_rows()` 按 `(regex_type, regex)` 去重，并合并 `normalized_tag_keys` 和 `sample_tags`。
3. `build_reorder_items()` 将这些正则行交给 AI 排序。
4. AI 返回排序后的 `ordered_ids`。
5. 最终 sheet 按 AI 排序后的正则行输出。

因此，`非译元素正则` sheet 应该是一行一个可执行正则，而不是一行一个原始 tag，也不是一行一个检测命中。

`非译元素正则` 字段：

| 字段 | 说明 |
|---|---|
| 排序 | AI 排序后的执行顺序，从 1 开始 |
| 正则表达式 | 最终用于匹配的裸正则，不带 `/regex/` 包裹 |
| 开始/结束/空 | 该正则属于开始标签、结束标签还是独立/自闭合/占位符 |
| 类型 | 非译元素类型，例如 HTML 代码、花括号占位符、方括号标签、变量、转义字符 |
| 非译元素例子 | 该正则覆盖的真实非译元素样例 |

提取术语时，输出这些 sheet：

1. `术语库`
2. `待审核`
3. `失败记录`
4. `非译元素正则`

默认不输出非译元素命中明细、保护后文本、预处理统计。这些内容只作为内部调试信息，不作为默认交付内容。

术语库和待审核 sheet 不再输出 `置信度` 字段。

### 阶段 7：WebUI 化

目标：

1. 在核心流程稳定后，将桌面可视化界面改造成 WebUI。
2. 后端能力保持模块化，避免业务逻辑写死在桌面 UI 中。
3. 配置项按“非译元素阶段”“术语召回阶段”“术语校验阶段”分组，方便后续前端表单化。
4. 运行进度、失败日志、导出结果路径要通过结构化状态提供给前端。

WebUI 实施原则：

1. WebUI 在所有核心流程稳定、测试覆盖足够后再开始，不与底层流程重构并行推进。
2. WebUI 只负责配置、启动、进度展示、结果查看和导出管理，不把业务规则写进前端。
3. 后端优先提供清晰的服务接口和状态对象，前端只消费结构化数据。
4. 页面文字、按钮、输入框、表格、进度提示都必须清晰可见，比例正常，不使用过小字号或拥挤布局。
5. 界面风格要简洁、美观、稳重，参考成熟工具型产品的设计方式，例如清晰的配置分组、明确的主按钮、可解释的状态提示、低噪声的信息层级。
6. 不追求花哨动效，优先保证长时间任务的可读性、可控性和错误可恢复性。
7. 第一版 WebUI 必须兼顾桌面浏览器使用，默认适配 1366×768、1440×900、1920×1080 三类常见窗口尺寸。
8. 移动端不是第一优先级，但窄屏下不能出现关键按钮不可见、文字重叠或横向滚动导致无法操作的问题。

推荐技术分层：

1. 后端保留现有 Python 核心能力，抽离成可被 Web 服务调用的任务服务。
2. Web 服务建议使用 FastAPI 或同等轻量框架，提供任务创建、任务状态、日志读取、结果下载、配置读写等接口。
3. 前端建议使用 React/Vue/Svelte 等成熟框架之一，优先选择团队后续维护成本低的方案。
4. 长任务进度可以使用轮询或 WebSocket；第一版可以先用稳定的轮询，后续再升级为 WebSocket。
5. 文件路径、导出路径、运行日志、失败记录不要只靠前端状态保存，必须由后端持久化。

WebUI 信息架构：

1. 首页 / 任务页：选择输入目录、选择文本列、选择运行模式、启动任务、查看当前任务状态。
2. 模型设置页：配置供应商、API Key、Base URL、模型名、并发、超时；模型仍然共用一套，但不同阶段可有独立行为配置。
3. 阶段设置页：分别配置非译元素阶段、术语召回阶段、术语校验阶段，例如字符数上限、是否开启 thinking、是否启用数值归一、单次候选输出策略。
4. 非译元素设置页：配置宽泛条目筛选正则、正则顺序、内置库开关、AI 发现/生成/排序开关。
5. 提取范围页：配置术语类型和启用状态，例如角色、场景、系统、道具、属性、技能、活动、UI。
6. 运行详情页：展示阶段进度、当前批次、成功/失败数、重试数、当前并发、最近日志、失败摘要。
7. 结果页：展示导出文件、术语库数量、待审核数量、失败数量、非译元素正则数量，并提供打开目录/下载文件入口。

关键交互要求：

1. 主流程按钮必须明确，例如“开始提取”“仅提取非译元素”“停止任务”“继续上次任务”。
2. 危险操作必须二次确认，例如停止任务、清空缓存、覆盖配置。
3. 长时间任务必须显示当前阶段，不允许只有一个无含义 loading。
4. 失败时必须给用户可操作信息，例如失败阶段、失败原因、可查看日志、可重试或导出失败记录。
5. 配置项要有简短说明，避免用户不知道该填什么。
6. 高级配置默认折叠，避免首屏压迫感。
7. API Key 输入框默认隐藏内容，并提供显示/隐藏切换。
8. 保存配置后要有明确反馈，例如“已保存”。
9. 任务完成后要突出显示输出文件位置和主要统计。
10. 所有表格必须有合理列宽、空状态提示和溢出处理。

视觉设计要求：

1. 字号层级清晰：正文不小于 14px，主要按钮和表单控件高度不应过小。
2. 按钮文字必须完整显示，不允许被截断。
3. 表单标签和输入框对齐，配置密集区域要用分组卡片或分栏降低阅读压力。
4. 使用克制的颜色系统，主色只用于主要操作、进度和强调信息。
5. 错误、警告、成功状态必须有明确视觉区分，但不要大面积刺眼颜色。
6. 日志区域使用等宽字体或清晰可读的日志样式，支持复制。
7. 页面整体留白要足够，不做拥挤面板。
8. 保持一致的圆角、阴影、边框、间距和按钮样式。
9. 图标可以使用，但不能依赖图标表达核心含义，必须有文字。
10. 优先参考成熟 SaaS / 开发者工具 / 数据处理工具的布局方式：左侧导航、顶部状态、主内容卡片、底部结果反馈。

Playwright 可用性测试要求：

1. WebUI 开发完成后必须使用 Playwright 进行自动化可用性测试，不能只靠人工点一遍。
2. Playwright 测试必须覆盖核心流程：打开页面、填写配置、选择模式、启动任务、查看进度、任务完成、查看导出结果。
3. Playwright 测试必须覆盖“仅提取非译元素”和“提取术语”两种模式。
4. Playwright 测试必须覆盖配置保存和重新加载，确认刷新页面后配置仍存在。
5. Playwright 测试必须覆盖失败状态展示，例如模拟 API 失败或 JSON 解析失败，确认页面显示失败原因和日志入口。
6. Playwright 测试必须覆盖停止任务和继续任务。
7. Playwright 测试必须检查关键按钮和文字是否可见，不允许出现按钮被遮挡、文字过小、文字溢出导致不可读。
8. Playwright 测试必须在至少一种标准桌面窗口尺寸下截图留档；推荐尺寸：1366×768 和 1440×900。
9. 每次明显修改 WebUI 布局后，都要重新运行 Playwright 测试。
10. Playwright 测试失败时，不允许仅更新截图或放宽断言，必须先判断是否真的存在可用性问题。

WebUI 验收标准：

1. 用户不阅读 README，也能根据界面完成一次基础提取任务。
2. 用户能清楚知道当前任务处于哪个阶段。
3. 用户能清楚知道任务是否成功、失败或被停止。
4. 用户能找到导出结果。
5. 用户能理解非译元素阶段、术语召回阶段、术语校验阶段分别有哪些配置。
6. 用户能在界面中区分普通配置和高级配置。
7. 所有主要文字和按钮在常见桌面分辨率下正常比例显示。
8. Playwright 核心流程测试全部通过。
9. 后端核心流程测试仍然通过，WebUI 不能破坏已有 pipeline。
10. WebUI 代码结构不能阻碍未来增加新阶段、新配置或新输出 sheet。

## 16. 已确认决议

### 16.1 流程决议

1. 占位 tag：采用内部安全格式，展示和导出时显示为 `<1>`、`<2>`。
2. 数值归一：仅在“移除数值后形成重复组”时触发。
3. 运行模式：只保留“提取术语”和“仅提取非译元素”。只要提取术语，就必然包含非译元素提取和保护流程。
4. 宽泛检测正则：用于筛出疑似包含非译元素的文本条目，不用于直接抠出最终非译元素。
5. 最终非译元素正则排序：永远交给 AI 排序，工具只校验 AI 排序结果是否合法。
6. 内置正则库：不在 AI 前抢先替换或拆分候选文本，只在 AI 发现非译元素后判断哪些元素已被现有规则覆盖。
7. 最终正则 AI 排序失败处理：最多重试 3 次；3 次失败后停止整个任务，并输出失败日志。
8. 仅提取非译元素时，只输出一张 `非译元素正则` sheet。
9. AI 非译元素召回时，允许省略没有非译元素的文本条目，不强制返回空数组。
10. 术语校验不使用 `confidence`，不使用 `risk_hints`，不使用 `risk_flags`。
11. 程序不做术语语义风险判断，不直接 hard approve；语义裁决交给 AI 提示词约束。

### 16.2 非译元素类型

类型参考 `D:\脚本\tag预处理\tag_ai_core.py` 的 `classify_tag()`，第一版使用：

1. `html`
2. `hash_brace`
3. `brace`
4. `bracket`
5. `at_var`
6. `dollar_var`
7. `percent_var`
8. `escape`
9. `html_entity`
10. `other`

open、close、empty 不作为类别，而作为正则规则字段。

## 17. 初步结论

这个方案的关键价值是：

1. 非译元素提取可以独立产出正则库。
2. 非译元素保护可以显著改善术语召回输入。
3. tag 和数值不再干扰重复文本判断。
4. 相同清洗文本只需召回一次，降低 token 和耗时。
5. 正则库会随着项目积累，越跑越省。
